# Rubric-Guided-Feedback-Generation-for-Automated-Essay-Assessment-in-Brazilian-Portuguese

# Automatic ENEM Essay Scoring with a local LLM (Llama 3.1) + BERTimbau

A system for **automatic scoring and evaluation of ENEM argumentative essays**
(ENEM is Brazil's national high-school exam), built on top of the **Extended Essay-BR**
corpus. The project explores three fronts:

1. **Producing the SCORE** of the 5 competencies (C1–C5) through two independent paths:
   - **Path A — LLM:** *Llama 3.1 8B* running **locally via Ollama**, with *few-shot*
     examples from Essay-BR itself and **self-consistency** (median of several runs).
   - **Path B — Encoder:** *BERTimbau large* with **fine-tuning** (regression of the 5 scores).
2. **Writing explanatory FEEDBACK** per competency (Llama justifies each score).
3. **Evaluating feedback quality** with an **LLM as a judge**: in a **blind A/B comparison**,
   an impartial judge decides whether the better feedback came from the **human** or the **model**.

> The whole pipeline was designed to run on a **personal machine** (reference: an
> **NVIDIA RTX 3060 12 GB**), with the LLM running **100% locally** (no API cost). The
> Gemini judge is optional and uses the free tier.

## Step-by-step installation

### 1. Clone the repository

```bash
git clone https://github.com/<your-username>/essay-br.git
cd essay-br
```

### 2. Create and activate a virtual environment

```bash
# Windows (PowerShell)
python -m venv .venv
.\.venv\Scripts\Activate.ps1

# Linux / macOS
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Install the Python dependencies

> ⚠️ The `requirements.txt` in this repository lists only `pandas` and `scikit-learn`
> (enough to *read* the corpus). To run the **full** pipeline, install everything:

```bash
# PyTorch with CUDA 12.1 (to use the RTX 3060)
pip install torch==2.5.1 --index-url https://download.pytorch.org/whl/cu121

# Remaining libraries
pip install transformers==4.46.3 accelerate==1.13.0 \
            ollama==0.6.2 pandas==2.3.1 scikit-learn==1.7.1 numpy==2.3.1

# Optional — only if you will use the Gemini judge
pip install google-genai==2.9.0
```

> Without an NVIDIA GPU, replace the first line with `pip install torch==2.5.1`
> (CPU build). BERTimbau training will be slow, but inference still works.

### 4. Install Ollama and pull Llama 3.1 8B

1. Install the **Ollama** app: <https://ollama.com/download>
2. Pull the model and keep the service running:

```bash
ollama pull llama3.1:8b
ollama serve     # keeps the local server alive (starts automatically on Windows/Mac)
```

### 5. (Optional) Gemini key for the alternative judge

Get a free key at [Google AI Studio](https://aistudio.google.com/) and export it:

```bash
# Windows (PowerShell)
$env:GEMINI_API_KEY = "your_key"
# Linux / macOS
export GEMINI_API_KEY="your_key"
```

### Data format

- **`extended_essay-br.csv` / splits** — columns `prompt, title, essay, c1, c2, c3, c4, c5, score`.
  `essay` is a **list of paragraphs** (read with `eval` in `build_dataset.py`); each `c*`
  is the human competency score (0–200) and `score` is their sum (0–1000).
- **`feedback_humano.csv`** — `prompt, essay_text, specific_c1..c5, llm_c1..c5`.
  `specific_c*` = **human** specific feedback; `llm_c*` = feedback generated by the **model**.
- **`dataset_c{n}.csv`** — `prompt, essay_text, specific_c{n}, llm_c{n}, entrada_A, entrada_B, random, escolha_llm`.
  For each essay, the human and model feedback are shuffled into A/B; the `random` column
  records **which source went to which position** (e.g. `entrada_A=llm_c1; entrada_B=specific_c1`)
  and `escolha_llm` stores the judge's decision (`A` or `B`).

## How to use each script (and what it produces)

> Run everything **from inside the `essay-br` folder** and with **Ollama active**.

### 0. Generate the splits (optional — they already ship ready)

```bash
python build_dataset.py
```

Creates `extended-corpus/splits/{train,dev,test}.csv` with a **70/15/15 split stratified
by score** (`random_state=230`, therefore **deterministic**: everyone reproduces exactly
the same splits). Prints the `head()` of the test set.

---

### Path A — Score via LLM

#### `corretor_redacao5.py` — 100% LLM grader (interactive)

```bash
python corretor_redacao5.py
```

You paste the **prompt** (topic) and the **essay**; the program waits for `FIM` on a line
by itself to end each input. Internally it:

- builds **4 few-shot examples** drawn from the *training split* (real Essay-BR scores);
- forces the model to **justify before scoring** and to follow a **scoring-band rubric**;
- grades **K=3 times** and uses the **median** of the scores (self-consistency) — reduces noise.

**Output:** C1–C5 scores, a justification per competency, the **final score (0–1000)** and
general feedback. The model (`llama3.1:8b`) and `K` are configurable at the top of the file.

#### `avaliar_metricas_1.py` — metrics of the LLM grader

```bash
python avaliar_metricas_1.py
```

Runs `corretor_redacao5` over **N=30** essays from the **validation split** and prints,
**per competency**: Accuracy (exact band), Precision/Recall/F1, **±1 band** accuracy,
**MAE** (points) and **QWK** (ordinal agreement — the field's standard metric), plus the
**final-score MAE**. Adjust `N` at the top (each essay = K LLM calls, so it's slow). Use it
to **compare changes** in the prompt/few-shot: higher QWK and lower MAE = improvement.

#### `ver_exemplos.py` — utility to build few-shot

```bash
python ver_exemplos.py
```

Prints **3 real essays** (LOW, MEDIUM and HIGH band) with their human scores, so you can
read them and write reference feedback (used to populate the *few-shot* examples).

---

### Path B — Score via BERTimbau

#### `treinar_bertimbau.py` / `treinar_bertimbau_2.py` — fine-tuning

```bash
python treinar_bertimbau_2.py     # recommended (has a GradScaler compatibility fallback)
```

Fine-tunes `neuralmind/bert-large-portuguese-cased` as a **5-output regressor** (one per
competency, normalized to [0,1] and then mapped to the nearest {0,40,…,200} band).
Configuration that fits the **RTX 3060 12 GB**: `MAXLEN=512`, `BATCH=4`, `ACCUM=2`
(effective batch 8), `LR=1e-5`, **fp16 (AMP)**, **5 epochs**.

**Output:** at each epoch it prints **QWK and MAE per competency** on validation and **saves
the best epoch** to `bertimbau_redacao.pt` (+ tokenizer in `bertimbau_redacao_tok/`).
> The two files are nearly identical; `_2` just tolerates different PyTorch versions when
> creating the `GradScaler`. **Prefer `_2`.**

> 💾 **Don't want to train?** Download the **already-trained** model and place the
> `bertimbau_redacao.pt` file in the **project root** (the tokenizer
> `bertimbau_redacao_tok/` already ships in the repo). Then you can run `avaliar_test.py`
> and the `corretor_completo*.py` scripts directly, without fine-tuning.
>
> - Model page (~1.3 GB): <https://huggingface.co/LuanMez/Treinamento_Bertimbau/blob/main/bertimbau_redacao.pt>
> - Direct download from the terminal:
>   ```bash
>   curl -L -o bertimbau_redacao.pt https://huggingface.co/LuanMez/Treinamento_Bertimbau/resolve/main/bertimbau_redacao.pt
>   ```

#### `avaliar_test.py` — final numbers on the test set

```bash
python avaliar_test.py
```

Loads the saved model and evaluates on the **test split** (essays never seen and **not**
used to pick the best epoch). Prints **MAE and QWK per competency**, **final-score MAE**
and **mean QWK**. It trains nothing — it requires `bertimbau_redacao.pt` and
`bertimbau_redacao_tok/` to be present (already included in the repository).

#### `kfold_bertimbau.py` — robust cross-validation

```bash
python kfold_bertimbau.py                 # K=5, 4 epochs per fold (default)
python kfold_bertimbau.py --k 10 --epochs 5
python kfold_bertimbau.py --avaliar-test  # at the end, retrains on the pool and evaluates the test
python kfold_bertimbau.py --salvar-melhor # saves the best fold to bertimbau_kfold_best.pt
python kfold_bertimbau.py --seed 99       # reproducibility with a different seed
```

Runs a **stratified K-Fold** (pool = train+dev; test is held out). It trains each fold from
scratch and reports **mean, standard deviation, 95% CI (t-Student), minimum and maximum** of
QWK and MAE per competency — i.e. **statistically robust** numbers with a stability range
across folds. This is the **heaviest** experiment (it trains K models).

### Integrated system — Score (BERTimbau) + Feedback (Llama)

The core idea: **each model on the task it is best at**. BERTimbau gives the **scores**
(more accurate and cheaper); Llama writes the **feedback** consistent with them **without
changing them**.

#### `corretor_completo.py` and `corretor_completo_3_original.py` (interactive)

```bash
python corretor_completo_3_original.py    # version with feedback few-shot (more consistent)
python corretor_completo.py               # lean version (no few-shot in the feedback)
```

You paste prompt + essay (ending with `FIM`). The flow:
1. **BERTimbau** predicts the 5 scores;
2. **Llama** receives the prompt, the essay and **the already-fixed scores** and writes the
   justifications + general feedback, forbidden from changing the scores.

The difference between the two: `corretor_completo_3_original.py` includes **3 few-shot**
feedback examples (low/medium/high essays with model justifications), which makes the tone
and format more stable. **Requires the trained BERTimbau** and Ollama active.

---

### Feedback evaluation — LLM as judge

Measures whether the **model's** feedback is as good as the **human's**, in a **blind** comparison.

#### `juiz_llm.py` (judge = Llama) or `juiz_llm_gemini.py` (judge = Gemini)

```bash
python juiz_llm.py                  # judges dataset_c1..c5.csv in the current folder (Llama judge)
python juiz_llm.py datasets/ 3      # judges only dataset_c3.csv inside datasets/

python juiz_llm_gemini.py           # same task, Gemini judge (needs GEMINI_API_KEY)
```

For each row, the judge reads the prompt, the essay and the **two feedbacks (A/B)** about
that competency and picks the better one by **accuracy → specificity → usefulness →
coherence**, filling the **`escolha_llm`** column (`A`/`B`). Characteristics:

- **blind comparison:** the judge does not know which feedback is human and which is the model's;
- **saves after each row and resumes where it stopped** (skips already-judged rows);
- **Gemini** belongs to a **different family** from the Llama that *generated* the feedback,
  which **mitigates self-preference bias** — that's why it is the recommended judge for the
  headline number.

> ⚠️ The `dataset_c*.csv` files **already ship with the `escolha_llm` column filled in**.
> Since the scripts resume where they stopped, running again **redoes nothing**. To
> reproduce from scratch, **clear the values in the `escolha_llm` column** (keep the header)
> before running.

#### `apurar_escolhas.py` — tally (reliable result)

```bash
python apurar_escolhas.py                       # generates apuracao_escolhas.csv
python apurar_escolhas.py datasets/ resumo.csv  # custom folder and output name
```

Cross-references `escolha_llm` (chosen A/B) with `random` (which source was in each
position) and computes, **per competency and overall**, the percentage of times the judge
preferred the **human (`specific`)** feedback or the **model (`llm`)** feedback. Generates
**`apuracao_escolhas.csv`**.

#### `analise_resultados.py` — alternative tally

```bash
python analise_resultados.py        # generates resumo_juiz.csv
```

Does a similar count and also reports **position bias** (A vs. B) and, if a human-judge
column exists, **agreement + Cohen's kappa**. Generates **`resumo_juiz.csv`**.

> **Mind the format:** this script was written for a `random` column that names **a single
> source**. In the included datasets, `random` names **both positions**
> (`entrada_A=…; entrada_B=…`), which always contain both markers (`llm` and `specific`).
> Therefore, **for these files the correct tally is the one from `apurar_escolhas.py`**
> (`apuracao_escolhas.csv`); `resumo_juiz.csv` ends up reflecting position bias, not the
> source. Use `analise_resultados.py` only if you standardize `random` to a single source.

## Results

### Feedback evaluation (LLM as judge, blind comparison)

Consolidated result in **`apuracao_escolhas.csv`** — 164 essays × 5 competencies =
**820 blind comparisons**. How often the judge preferred the **model (`llm`)** feedback vs.
the **human (`specific`)** one:

| Competency | Preferred **LLM** | Preferred **human** |
|:----------:|:-----------------:|:-------------------:|
| C1 — Standard grammar          | **71.95%** | 28.05% |
| C2 — Comprehension/structure   | **76.22%** | 23.78% |
| C3 — Argumentation/repertoire  | **63.41%** | 36.59% |
| C4 — Cohesion                  | **88.41%** | 11.59% |
| C5 — Intervention proposal     | **83.54%** | 16.46% |
| **OVERALL**                    | **76.71%** | **23.29%** |

➡️ **In ~77% of the blind comparisons, the model-generated feedback was judged better than
the human specific feedback** — an advantage in every competency, strongest in cohesion (C4)
and intervention proposal (C5).

### Score evaluation (BERTimbau and LLM)

The metrics (**QWK** and **MAE** per competency, final-score MAE) are **printed to the
terminal** when you run `avaliar_test.py`, `avaliar_metricas_1.py` and `kfold_bertimbau.py`.
Since they depend on your machine and seed, **run the scripts to see the numbers in your own
environment** — the output format (printed in Portuguese by the scripts) is:

```
=== TESTE — N redações (números finais) ===
  C1:  MAE=...   QWK=...
  ...
  NOTA FINAL: MAE=...   |   QWK médio: ...
```

> **QWK** (Quadratic Weighted Kappa) is the reference metric for *Automatic Essay Scoring*:
> it measures **ordinal** agreement with the human graders of Essay-BR.

## Notes and limitations

- **Essay-BR does not include the full topic statement**, only the essay **title**. The
  scripts use the title as the "prompt"; this is enough to measure the scores, but it is
  **not ideal for evaluating off-topic essays (C2)**.
- **The metrics reflect agreement with the Essay-BR graders**, which may differ from a
  specific teacher's standard.
- **VRAM:** running BERTimbau and Ollama together may exhaust GPU memory. In
  `corretor_completo.py` you can force BERTimbau onto the CPU (`DEVICE = "cpu"`) — single-essay
  inference is fast either way.
- **Reproducibility:** the splits (`random_state=230`) and the Gemini judge
  (`temperature=0.0`) are deterministic; Llama uses `temperature` > 0 in some scripts, so
  small variations between runs are expected (self-consistency reduces this).
- `requirements.txt` only covers reading the corpus; use the full installation block from
  the [Installation](#step-by-step-installation) section.

## Corpus reference

This project extends the **Essay-BR** corpus, described
[in this paper](https://sol.sbc.org.br/index.php/dsw/article/view/17414): essays written by
Brazilian high-school students, graded by professionals according to ENEM criteria.
